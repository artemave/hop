# hop session model and command contract

The current `hop` contract is a session-oriented CLI built around projects, Sway workspaces, Kitty windows, and one shared Neovim instance.

Per [hop_spec.md](../../hop_spec.md), a session corresponds to a project root and consists of a dedicated Sway workspace named `p:<session_name>`, one shared Neovim instance, multiple Kitty terminal windows identified by role, and an optional browser window. The intended command surface is `hop`, `hop switch <session>`, `hop list`, `hop edit [target]`, `hop term --role <name>`, `hop run [--role <name>] "<command>"`, and `hop browser [url]`.

The current Python implementation is a parser plus adapter-driven command runner with session entry, role-routing, and Sway primitives wired end to end. `hop.session` centralizes project-root discovery, session-name derivation from the project directory basename, and workspace-name derivation as `p:<session>`. `hop.commands.session` owns enter, switch, and list orchestration: `hop` resolves the current project session, switches to its workspace, and asks the terminal adapter to ensure the shared `shell` terminal exists; `hop switch <session>` targets `p:<session>` directly; and `hop list` prints live session names discovered from Sway workspaces without the `p:` prefix in sorted order.

`hop.commands.term` and `hop.commands.run` now extend that session routing model to Kitty role terminals. `hop term --role <name>` resolves the current project session, switches to its workspace, and focuses the Kitty terminal for that role or creates it when missing. `hop run [--role <name>] "<command>"` resolves the same session context, defaults the role to `shell`, creates the target role window when needed, and injects the exact command string plus a trailing newline into it without refocusing an already-open role terminal. The command is fire-and-route: `hop` returns after dispatch and does not proxy the child command's exit status.

`hop.sway` provides the default Sway adapter via direct IPC over `SWAYSOCK` rather than shelling out to `swaymsg`. `hop.kitty` now provides the default Kitty adapter via Kitty's native remote control protocol, using the controlling TTY when `hop` runs inside Kitty and `KITTY_LISTEN_ON` when it must talk over a socket. Role windows are created as OS windows rooted at the project directory and tagged with stable session/role metadata in Kitty environment and user-variable fields so repeated `hop term` and `hop run` calls reuse the same `shell`, `test`, `server`, or other role window instead of creating duplicates. The repository now documents the intended external consumer flow in [docs/vigun.md](../../docs/vigun.md), which pins `hop run --role test "<command>"` as the stable test-execution entrypoint for `vigun`. `hop.app` routes session, terminal, and run commands through those adapter boundaries alongside the still-separate Neovim and browser integrations.
