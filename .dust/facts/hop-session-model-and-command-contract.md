# hop session model and command contract

The current `hop` contract is a session-oriented CLI built around directories, Sway workspaces, Kitty windows, and one shared Neovim instance.

Per [hop_spec.md](../../hop_spec.md), a session corresponds to the exact directory where `hop` is invoked and consists of a dedicated Sway workspace, one shared Neovim instance, multiple Kitty terminal windows identified by role, and an optional browser window. Repeating a command from the same directory targets the same session, while invoking `hop` from a nested directory targets a distinct nested session. The intended command surface is `hop`, `hop switch <session>`, `hop list`, `hop edit [target]`, `hop term --role <name>`, `hop run [--role <name>] "<command>"`, `hop tail <run-id>`, `hop browser [url]`, and `hop kill`.

**Session identity is directory-rooted.** `hop.session` treats the provided directory itself as the session root, derives the session name from that directory basename, and derives the workspace name as `p:<full_absolute_path>` (e.g. `p:/home/user/projects/myapp`). This ensures two directories with the same basename — such as `/path/a/myapp` and `/path/b/myapp` — are always distinct sessions with distinct Sway workspaces. Kitty terminal windows and the shared Neovim instance are matched by the stored `HOP_PROJECT_ROOT` path (written into Kitty environment and user-variable fields at launch time), not by session name alone. The Neovim socket address is derived from a SHA-256 hash of the project root path, guaranteeing per-directory uniqueness.

`hop.commands.session` owns enter, switch, list, and ad-hoc-spawn orchestration: `hop` resolves the current directory session, switches to its workspace (`p:<full_path>`), and asks the terminal adapter to ensure the shared `shell` terminal exists; `hop switch <session>` looks up active Sway workspaces matching that basename and switches to the first alphabetically — if no match exists it raises an error; `hop list` enumerates live Sway workspaces with the `p:` prefix and (alphabetical) prints either the names alone or, with `--json`, JSON records `{name, workspace, project_root}` — `project_root` is looked up from `hop.state.load_sessions()`, which reads per-session JSON files persisted at bootstrap to `${XDG_RUNTIME_DIR}/hop/sessions/<name>.json` (env override `HOP_SESSIONS_DIR`). Bare `hop` (and `hop term` without `--role`) parses to a single `EnterSessionCommand`. At dispatch time, `hop.app.execute_command` resolves the cwd-derived session and asks Sway for the focused workspace; if it matches `session.workspace_name`, it routes to `hop.commands.session.spawn_session_terminal`, which lists existing role windows for the session, picks the next free `shell-<N>` role (starting from `shell-2`), and launches a new Kitty role window with that role (no workspace switch — by definition we're already on the right one). Otherwise it routes to the regular `enter_project_session` flow that switches to `p:<session>` and ensures the canonical `shell` terminal exists.

`hop.commands.edit` now extends that same session-routing model to the shared editor. `hop edit` resolves the current invocation directory session, ensures the single session Neovim instance exists, and focuses it. `hop edit <target>` routes file and file-plus-line targets into that same Neovim instance instead of spawning a separate editor process.

`hop.commands.term` and `hop.commands.run` now extend that session routing model to Kitty role terminals. `hop term --role <name>` resolves the current invocation directory session and focuses the Kitty terminal for that role or creates it when missing. `hop run [--role <name>] "<command>"` resolves the same session context, defaults the role to `shell`, creates the target role window when needed, and injects the exact command string plus a trailing newline into it without refocusing an already-open role terminal. `hop run` also generates an opaque run id, persists `{window_id, session, role, dispatched_at}` to `${HOP_RUNS_DIR or XDG_RUNTIME_DIR/hop/runs}/<id>.json`, and prints the id to stdout — it still returns immediately without waiting for the dispatched command. `hop.commands.tail` consumes that run state: `hop tail <run-id>` polls the role window's `at_prompt` flag (Kitty's serialization of OSC 133 prompt boundaries) and, once the shell is back at a prompt, fetches `last_cmd_output` for that window via Kitty remote control and writes it to stdout. `hop tail` exits 0 on successful delivery and does not propagate the inner command's exit status; it requires Kitty shell integration to be active in the role terminal.

`hop edit`, `hop term`, `hop run`, and `hop browser` no longer issue a Sway workspace switch — they assume the caller is already on `p:<session>`, which holds whenever the command is invoked from one of that session's terminals. Workspace entry remains the responsibility of bare `hop` (which switches as part of entering or creating the session) and `hop switch <name>`. Removing the pre-switch eliminates a multi-monitor race where Kitty's `launch --type=os-window` and Sway's focus change were resolved out of order, occasionally placing the new role window on the wrong workspace.

`hop.sway` provides the default Sway adapter via direct IPC over `SWAYSOCK` rather than shelling out to `swaymsg`. It now also exposes tree inspection plus focus, move, and mark helpers so other adapters can manage OS windows through native Sway criteria. `hop.kitty` provides the default Kitty adapter via Kitty's native remote control protocol, addressing each session's Kitty process by a deterministic socket address `unix:@hop-<session_name>`. The first window for a session bootstraps the Kitty process via `subprocess.Popen(["kitty", "--listen-on", addr, "--directory", project_root, "--title", ..., "--name", ..., "--override", "allow_remote_control=yes"])` with `HOP_*` environment variables set; subsequent windows are added via `kitty @ launch --type=os-window` against the same socket. Role windows are created as OS windows rooted at the session root directory and tagged with stable session/role metadata in Kitty environment and user-variable fields so repeated `hop term` and `hop run` calls reuse the same `shell`, `test`, `server`, or other role window instead of creating duplicates. Because the socket address is derivable from the session name, hop can reach Kitty from sway-exec contexts (keybinding wrappers, scripts) without `KITTY_LISTEN_ON` plumbing. `hop.editor` applies the same native-control pattern to the shared Neovim lifecycle: it launches the editor in its own Kitty OS window with stable session/editor metadata, binds Neovim to a deterministic per-session `--listen` socket under the runtime directory, reuses that server for later `hop edit` calls, and recreates it cleanly after `:qa` when the socket stops responding. `hop.browser` now completes the session surface for `hop browser [url]`: it resolves the system default browser desktop entry, launches a fresh default-browser window when the session has no browser yet, tags that window with a stable session-specific Sway mark, focuses the marked window on repeated calls, and moves it back to `p:<session>` if the user previously dragged it elsewhere. `hop.commands.browser` mirrors the other session-scoped commands by resolving the invocation directory into a session and delegating browser reuse or URL opening to the browser adapter. The repository now documents the intended external consumer flow in [docs/vigun.md](../../docs/vigun.md), which pins `hop run --role test "<command>"` as the stable test-execution entrypoint for `vigun`. `hop.app` now routes edit, session, terminal, run, browser, and kill commands through those adapter boundaries. `hop kill` resolves the session from the caller's exact current working directory, discovers all managed windows (Kitty role terminals, the shared editor window identified by `hop_project_root` metadata, and the marked session browser via `hop_browser:<session_name>` Sway mark even if it drifted to another workspace), closes them all, and removes the session workspace via `workspace back_and_forth` if it still exists after teardown. Non-managed windows on the workspace are not closed.
