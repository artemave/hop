# hop target dispatch and behavior guarantees

The current `hop` contract includes visible-output target dispatch and strict window reuse guarantees.

Per [hop_spec.md](../../hop_spec.md), users must be able to select file references or URLs from visible Kitty output and dispatch them to the shared Neovim instance or the session browser. The current implementation uses Kitty's native `hints` interface with custom processing in `kittens/open_selection/main.py`. The required target patterns are file paths, file paths with line numbers, git diff paths with `a/` or `b/` prefixes, URLs, and Rails-style references. File resolution tries absolute paths first, then the terminal working directory, then the project root, strips git diff prefixes before the fallback lookup, and ignores unresolved targets.

The current terminal-routing contract now enforces reuse by role within a session. `hop term --role <name>` focuses the existing Kitty OS window for that session/role pair or creates it if it does not exist. `hop run [--role <name>] "<command>"` defaults to the `shell` role, creates the missing role window if needed, injects the command text plus a trailing newline into that exact role window, and returns once routing succeeds instead of waiting on the child process. The Kitty adapter prevents duplicate role windows during repeated calls by listing live Kitty windows through the native remote control protocol and matching stable hop metadata stored on each managed window rather than relying on transient visible titles or raw window IDs.

The editor-routing contract now applies the same reuse guarantee to the editor. Focus / launch live on `hop term --role editor`, which (alongside `hop open <target>` for file/Rails-ref dispatches) checks a deterministic per-session Neovim server address first, launches a new editor window only when that server is missing, focuses the existing editor window when it is already running, and sends `path` or `path:line` targets into the shared instance via Neovim remote commands. Editor focus is performed through Sway IPC by looking up the editor's `_hop_editor:<session>` mark (with leading underscore so Sway hides it from window titles); on first sighting hop discovers the unmarked Kitty OS window via `app_id == "hop:editor"` on `p:<session>` and tags it. Window names no longer carry the session name — disambiguation lives in the mark. This mirrors the browser path, so `hop term --role editor`, `hop open <target>`, and kitten-driven file dispatches all switch the Sway workspace to the editor when it lives elsewhere instead of relying on Kitty's `focus-window` to escalate. If the user exits Neovim with `:qa`, the stale socket is discarded and the next `hop term --role editor` (or any `hop open <file>`) recreates the editor instead of accumulating duplicate windows.

`hop open` and the open-selection kitten share a single target parser (`hop.targets.parse_visible_output_target` plus `resolve_target`) and a single dispatcher (`hop.commands.open.dispatch_resolved_target`). Adding a new target shape (e.g. a new framework's `Foo::BarController#baz` ref or a different traceback pattern) means editing one resolver branch; both code paths pick it up for dispatch automatically. The kitten layers two pieces on top of the shared core that the CLI does not need: its own discovery regex `VISIBLE_OUTPUT_TARGET_PATTERN` (used by `mark()` to find candidate spans in arbitrary terminal output for highlighting — adding a new target type that is not already path-shaped here requires a parallel edit), and an existence filter (`backend.paths_exist` against the focused session) to suppress dispatches for path-shaped tokens that do not exist in the backend's filesystem namespace. The CLI deliberately skips the existence check so `hop open` of a not-yet-created file lands in an `:enew`-style buffer rather than failing silently. `hop open` requires a target — focusing the editor lives on `hop term --role editor`.

The browser-routing contract now applies the same reuse guarantee to `hop browser [url]`. `hop browser` launches a new system-default browser window only when the session does not already own one, then marks that Sway container with stable session browser metadata so later calls can rediscover it exactly. `hop browser <url>` focuses that marked window before delegating the URL to the default browser, so `hop` routes browser work to the session-owned window instead of adopting unrelated browser windows. If the marked browser was moved off the session workspace, the next `hop browser` reattaches it to `p:<session>` rather than silently switching to some other browser window.
