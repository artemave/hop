# hop target dispatch and behavior guarantees

The current `hop` contract includes visible-output target dispatch and strict window reuse guarantees.

Per [hop_spec.md](../../hop_spec.md), users must be able to select file references or URLs from visible Kitty output and dispatch them to the shared Neovim instance or the session browser. The required target patterns are file paths, file paths with line numbers, git diff paths with `a/` or `b/` prefixes, URLs, and Rails-style references. File resolution tries absolute paths first, then the terminal working directory, then the project root, and ignores unresolved targets.

The current terminal-routing contract now enforces reuse by role within a session. `hop term --role <name>` focuses the existing Kitty OS window for that session/role pair or creates it if it does not exist. `hop run [--role <name>] "<command>"` defaults to the `shell` role, creates the missing role window if needed, injects the command text plus a trailing newline into that exact role window, and returns once routing succeeds instead of waiting on the child process. The Kitty adapter prevents duplicate role windows during repeated calls by listing live Kitty windows through the native remote control protocol and matching stable hop metadata stored on each managed window rather than relying on transient visible titles or raw window IDs.

The editor-routing contract now applies the same reuse guarantee to `hop edit`. `hop edit` checks a deterministic per-session Neovim server address first, launches a new editor window only when that server is missing, focuses the existing editor window when it is already running, and sends `path` or `path:line` targets into the shared instance via Neovim remote commands. If the user exits Neovim with `:qa`, the stale socket is discarded and the next `hop edit` recreates the editor instead of accumulating duplicate windows.
